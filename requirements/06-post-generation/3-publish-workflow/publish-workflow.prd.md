# PRD-06.3.01: Publish Workflow

**Epic:** Publish Workflow (EPC-06.3) — Post Generation (DOM-06)
**Status:** Approved
**Created:** 2026-03-21

---

## Problem Statement

After a post is created and categorized, its final status needs to be set and the site owner needs to know the outcome. Site owners who prefer to review posts before they go live need draft behavior; those who want hands-off automation need immediate publishing. In both cases, a clear admin-facing signal after a cron run tells the site owner what happened without requiring them to hunt through the posts list.

---

## Target Users

- **Site owners** — want the post status to match their configured preference without manual intervention; want to know at a glance what the last cron run produced
- **The plugin pipeline** — this is the final step before notification; it determines the post's visibility and fires the notification trigger
- **Developers** — want a filter hook to override post status on a per-release basis

---

## Overview

Reads the effective post status for the given repository (per-repo setting with global fallback) and updates the post to that status. For published posts, sets the publication date to the current time. Exposes a filter hook for per-release status overrides. After all posts in a cron run are processed, displays a single admin notice summarizing how many posts were drafted or published, with edit links to each.

---

## User Stories & Acceptance Criteria

### US-001: Apply the configured post status

As the plugin pipeline, I want the post status set to the site owner's configured preference so the post is immediately visible (or held for review) without any manual action.

**Acceptance Criteria:**

- [ ] **AC-001:** The effective post status is determined by reading the per-repo default (PRD-03.2.01), falling back to the global default (PRD-03.2.02) if no per-repo value is set.
- [ ] **AC-002:** If the effective status is `publish`, the post is published with the publication date set to the current time.
- [ ] **AC-003:** If the effective status is `draft`, the post remains a draft. No publication date is set.
- [ ] **AC-004:** "Generate draft now" (PRD-03.2.01) always results in a `draft` post regardless of the configured default. The publish workflow layer respects this override when it is passed.

---

### US-002: Allow per-release status override via filter

As a developer, I want to override the post status for a specific release programmatically, so I can implement custom publishing logic (e.g., auto-publish only major releases).

**Acceptance Criteria:**

- [ ] **AC-005:** A filter hook is applied to the resolved post status before it is set, receiving the status string, the post ID, the repo identifier, and the release tag.
- [ ] **AC-006:** The filter return value is used as the final post status. Any valid WordPress post status string returned by the filter is accepted.

---

### US-003: Surface the outcome of each cron run in wp-admin

As a site owner, I want to see a summary of what the last scheduled run produced so I know whether to expect new drafts to review or live posts to have been published.

**Acceptance Criteria:**

- [ ] **AC-007:** After a cron run processes one or more releases, a WordPress admin notice is displayed the next time a user with `manage_options` visits wp-admin. The notice shows:
  - The number of posts drafted (if any), each with a direct edit link
  - The number of posts published (if any), each with a direct edit link
  - The number of releases skipped due to errors (if any), with a prompt to check the debug log
- [ ] **AC-008:** If a cron run processes no new releases (nothing new since the last run), no admin notice is displayed.
- [ ] **AC-009:** The admin notice is dismissible. Once dismissed, it does not reappear for the same cron run's results.
- [ ] **AC-010:** Admin notices from multiple cron runs (e.g., if the site owner did not log in between runs) are combined or the most recent run's notice is shown — notices do not stack unboundedly.

---

## Business Rules

- **BR-001:** Post status is set as the last step of the post generation pipeline, after taxonomy assignment (EPC-06.2). Publishing a post before its terms are assigned is not permitted.
- **BR-002:** The publish workflow step triggers the notification pipeline (DOM-07) after updating the post status. Notifications are sent based on the final status and the site owner's notification trigger preference.
- **BR-003:** Onboarding preview drafts (PRD-04.2.01) bypass this layer — they are always created as drafts directly in the post creation step. The publish workflow only applies to posts generated by automated cron runs and "Generate draft now" triggers.
- **BR-004:** Post dates are always the current time. There is no backdating to the GitHub release date.

---

## Out of Scope

- Scheduled/future publishing (v1 only supports immediate draft or publish)
- Email notification content and delivery (DOM-07 — triggered by this layer but not defined here)
- Taxonomy assignment (EPC-06.2 — must complete before this step runs)
